MITRE ATT&CKT1195.001

Your organization's CI/CD pipeline has ingested a build that includes a widely-used open-source library. Security tooling has flagged anomalous outbound connections originating from a production service. Initial triage suggests the library version pulled last night contains malicious code inserted via a compromised maintainer account. The artifact has potentially been deployed to multiple environments. You do not yet know the full blast radius.

P-01 DETECTION & INITIAL TRIAGE IDENTIFY ▼
Initial Discovery
✓
Identify the specific package(s), library name(s), and version(s) flagged or suspected
Record exact package name, registry (npm, PyPI, Maven, NuGet, etc.), and version string. Hash the artifact (SHA-256).
CRITICAL
✓
Determine the initial detection vector — was this caught by SCA tooling, EDR, SIEM alert, external advisory (NVD, OSV, vendor), or manual discovery?
DISCUSSION: What is our standard detection path for malicious packages? Is it reactive or proactive?
CRITICAL
✓
Confirm whether the malicious version was introduced via direct dependency or transitive (indirect) dependency
Pull dependency trees across all affected repos. Tools: npm ls, pip-tree, mvn dependency:tree, Snyk, Dependabot graph.
CRITICAL
✓
Establish the first time the malicious version was pulled into any environment — build logs, artifact repository timestamps, lock file commits
Check package-lock.json / poetry.lock / go.sum git history for when the version pin changed.
HIGH
💬 Discussion Prompt
How far back do our build logs and artifact pull records go? If the malicious version was ingested 60 days ago, could we reliably reconstruct the timeline — or does our log retention create a blind spot in the dwell time calculation?
💬 Discussion Prompt
Do we have a software bill of materials (SBOM) for affected services? If not, how long would it take us to generate one? Who owns that process?
Scope Assessment
✓
Enumerate all repositories, services, and applications consuming the compromised package
Search across all repos — monorepo, microservices, internal tooling, CI scripts, Docker base images, and lambdas.
CRITICAL
💬 Discussion Prompt
Do we have a centralised inventory of all software dependencies across every team and repo? If different teams manage their own dependency trees independently, how confident are we that our blast radius assessment is complete — and not just the services we know about?
✓
Determine which environments have executed code from the malicious build: dev, staging, production, CI/CD agents
CRITICAL
💬 Discussion Prompt
Are our dev and staging environments treated as lower-trust zones with restricted access to production secrets and data? Or do they share credentials, service accounts, or network segments with production — meaning a compromise in dev could still be a production-level breach?
✓
Assess data sensitivity of systems affected — PII, credentials, secrets, intellectual property, customer data
DISCUSSION: What is the worst-case data exposure scenario? Who needs to be looped in for breach assessment?
HIGH
✓
Analyze the malicious code — understand what it does (backdoor, credential harvest, crypto miner, data exfiltration, C2 beaconing)
Perform static analysis in an isolated sandbox. Map behavior to MITRE ATT&CK T1195.001, T1059, T1041.
CRITICAL
💬 Discussion Prompt
Do we have an isolated malware analysis sandbox available to reverse-engineer the malicious package? If we hand off to an external party for analysis, what data handling and NDA obligations apply — and could sharing the sample itself constitute a data disclosure?

P-02 CONTAINMENT ISOLATE ▼
Immediate Actions
✓
Block or quarantine the malicious package version in the artifact repository / package manager (Nexus, Artifactory, npm private registry, etc.)
CRITICAL
✓
Pin or lock all dependent packages to known-good versions in version control — freeze lockfiles and prevent automatic upgrades
Add temporary CI gate to reject builds consuming unapproved package versions.
CRITICAL
✓
Halt deployments to production until the dependency is remediated and builds are verified clean
DISCUSSION: What is the business impact of a deployment freeze? Who has authority to approve deployment exceptions?
CRITICAL
💬 Discussion Prompt
If a critical hotfix or regulatory-mandated patch needs to go out during the deployment freeze, what is our emergency exception process? Who can authorize it, and what additional verification steps are required before allowing that build through?
✓
Isolate affected production hosts or containers from the network if active exploitation/C2 activity is confirmed
HIGH
💬 Discussion Prompt
What is the operational impact of taking affected production containers offline — are there SLAs, customer-facing services, or safety-critical functions that would be disrupted? Do we have a pre-approved runbook for isolating services that weighs security response against availability obligations?
✓
Block outbound connections to identified C2 IP addresses, domains, or URLs at the perimeter firewall and endpoint level
Push IOC blocklists to proxy, firewall, DNS sinkhole, and EDR platform simultaneously.
HIGH
💬 Discussion Prompt
If the malware uses domain generation algorithms (DGA) or rotates C2 infrastructure frequently, blocking known IPs and domains is reactive at best. Do we have behavioural detections — e.g., unusual beacon intervals, JA3/JA3S TLS fingerprinting, or DNS entropy scoring — that can catch novel C2 channels we haven't seen yet?
Credential & Secrets Management
✓
Rotate all secrets, API keys, tokens, and credentials accessible to affected services — assume all were exfiltrated
Include: cloud provider creds, DB connection strings, JWT secrets, OAuth tokens, pipeline secrets, SSH keys.
CRITICAL
✓
Audit secrets manager (Vault, AWS Secrets Manager, Azure Key Vault) for access logs during the window of compromise
HIGH
💬 Discussion Prompt
Are secrets accessed by our applications injected at runtime or baked into environment variables at build time? If they're baked in, rotating the secrets manager value alone may not be sufficient — the old value may persist in container images, environment configs, or orchestration platform metadata. How do we ensure full rotation propagates everywhere?
✓
Revoke and reissue service account credentials and CI/CD pipeline tokens
DISCUSSION: Do our pipeline tokens follow least-privilege? Could the malicious package have used them to pivot?
HIGH
✓
Check build environment and CI agents for any persistence mechanisms planted during execution of the malicious code
HIGH
💬 Discussion Prompt
Are our CI/CD build agents ephemeral — spun up fresh for each build and destroyed afterward — or are they long-lived persistent VMs? If persistent, a backdoor planted during one build could silently affect every subsequent build. How do we verify agent integrity between runs?
💬 Discussion Prompt
If the attacker had full code execution during a build, what lateral movement paths are available? Can they reach production databases, cloud IAM, internal tooling, or source code repositories?

P-03 INVESTIGATION & FORENSICS ANALYZE ▼
Log Analysis
✓
Pull network flow / DNS / proxy logs for all hosts that ran the malicious package — look for beaconing, data staging, and exfiltration patterns
Time window: From first ingestion of malicious version to present. Check for DNS tunneling, HTTP POST to external IPs, encoded payloads.
CRITICAL
💬 Discussion Prompt
Do we have full-packet capture or only flow/metadata logs? If the exfiltration used HTTPS to a legitimate-looking CDN or cloud provider endpoint, would our proxy logs have the content or only connection metadata? What visibility gaps exist in our network logging stack that an attacker could exploit to exfiltrate data undetected?
✓
Review CI/CD pipeline execution logs for the affected builds — identify what commands ran, what files were created/modified, what network calls were made during build time
CRITICAL
💬 Discussion Prompt
How much detail do our pipeline execution logs capture? If a build step invoked a shell command or made an outbound call, would it appear in the log — or does our CI platform only log exit codes and step names? What would we need to enable to get command-level audit visibility during builds?
✓
Audit cloud provider access logs (CloudTrail, Azure Monitor, GCP Audit) for anomalous API calls from affected services during the compromise window
HIGH
💬 Discussion Prompt
Are cloud audit logs forwarded to a SIEM that is separate from the compromised environment — or do they live in the same cloud account that may have been accessed? If an attacker had sufficient IAM permissions, could they have tampered with or deleted CloudTrail/audit logs to cover their tracks before we started looking?
✓
Examine EDR/XDR telemetry for process executions, file writes, registry changes, and network connections spawned by the malicious dependency
MDE hunters: DeviceProcessEvents, DeviceNetworkEvents, DeviceFileEvents filtered to affected host + time window.
HIGH
Attack Chain Reconstruction
✓
Reconstruct the full attack timeline: package repository compromise → CI/CD ingestion → build execution → deployment → runtime exploitation
Map each stage to MITRE ATT&CK techniques. Document dwell time at each phase.
HIGH
💬 Discussion Prompt
When we map the dwell time across each stage — from initial ingestion to first detection — where is the largest gap? Was it in detection capability, alert triage time, or escalation and response speed? Which stage represents our biggest opportunity to shorten time-to-respond in a future incident?
✓
Investigate whether the threat actor achieved persistence — scheduled tasks, cron jobs, new service accounts, backdoored Docker images, Lambda layer injection
CRITICAL
💬 Discussion Prompt
If persistence was achieved inside a container image that has since been pushed to our registry and pulled by dozens of services, how do we systematically identify and quarantine every running instance? Do we have an image provenance and lineage tracking capability, or would we need to manually hunt across all running workloads?
✓
Determine if the compromise was targeted (your org specifically) or opportunistic (broad campaign against all consumers of the package)
DISCUSSION: Has this package compromise been publicly disclosed? Are peers in the industry affected? Should we share IOCs via ISAC?
MED
✓
Extract all IOCs: malicious package hash, C2 domains/IPs, exfiltration endpoints, dropped file hashes, registry keys, mutexes
HIGH
💬 Discussion Prompt
Once we have IOCs, what is our process for operationalising them — pushing to EDR, SIEM, firewall, and threat intel platform — within a time-bounded SLA? And do we track which IOCs have been deployed to which controls, so we can verify coverage without relying on assumption?
💬 Discussion Prompt
Our build environment runs with elevated permissions to access artifact registries and deploy to cloud. Could the attacker have used build-time execution to poison other artifacts or tamper with our release pipeline itself? How would we detect that?
✓
Check for dependency confusion or typosquatting as the initial vector — verify the package name against known internal package names
Confusion attack: attacker publishes a public package with the same name as your internal private package; resolver picks the higher version from public registry.
HIGH
💬 Discussion Prompt
If this was a dependency confusion attack, does our package manager configuration explicitly prefer our private registry over public sources — or does it fall back to public registries automatically? Have we audited every package manager config file across all repos for correct registry priority order?

P-04 ERADICATION & RECOVERY REMEDIATE ▼
Dependency Remediation
✓
Update or replace the compromised dependency with a verified clean version — confirm patch or fork availability
Options: Upgrade to patched version, vendor the dependency internally, fork and audit, remove entirely if non-essential.
CRITICAL
✓
Rebuild all affected artifacts from clean source using a freshly provisioned build environment — do not reuse potentially tainted build agents
CRITICAL
💬 Discussion Prompt
When we provision a "clean" build environment, what is our trust anchor — are we building from a known-good base image with a verified hash, or pulling the latest from a registry that could itself have been tampered with? How do we establish and verify a trusted baseline before treating a new build agent as clean?
✓
Run SCA (Software Composition Analysis) and SAST scans against the rebuilt artifacts before re-deployment
Tools: Snyk, OWASP Dependency-Check, Semgrep, Trivy, Grype.
HIGH
💬 Discussion Prompt
SCA tools rely on known vulnerability databases — they won't catch a novel malicious package that has no CVE yet. What additional verification steps do we apply beyond SCA to gain confidence in the clean build: code review, manual inspection of the replaced dependency, or behavioural testing in an isolated staging environment?
✓
Purge the malicious package version and all affected build artifacts from internal registries, caches, and artifact stores
HIGH
💬 Discussion Prompt
Beyond our primary artifact registry, where else might this package version be cached — developer workstations, local Docker layer caches, edge caches in our CDN, or CI agent workspace directories? How do we ensure the malicious version is not silently re-used from a stale local cache the next time someone runs a build?
Infrastructure Remediation
✓
Re-image or redeploy all hosts, containers, and serverless functions that executed the malicious code — do not trust in-place patching for confirmed compromised systems
CRITICAL
💬 Discussion Prompt
For stateful workloads — databases, message queues, caches — that cannot simply be re-imaged without data loss, what is our process for verifying their integrity? If a malicious process wrote data to a database or queue during the compromise window, how do we identify and cleanse those records without losing legitimate production data?
✓
Restore systems from pre-compromise backups where available and verified clean
DISCUSSION: Do our backups pre-date the first ingestion of the malicious package? What is the RTOs/RPO impact of restoring?
HIGH
✓
Audit and remove any persistence mechanisms identified during investigation (backdoor accounts, rogue cron jobs, unauthorized SSH keys, modified container base images)
CRITICAL
✓
Validate integrity of the CI/CD pipeline itself — confirm no tampering with pipeline configs, Dockerfile base images, GitHub Actions, or build scripts
HIGH
💬 Discussion Prompt
Do we pin GitHub Actions and third-party pipeline actions to specific commit SHAs rather than mutable tags like @main or @v2? If not, a compromised action maintainer could push malicious code under an existing version tag and our pipelines would silently adopt it on the next run — with no change to our own code. Is this on our hardening backlog?
💬 Discussion Prompt
How do we validate that our recovery is complete? What evidence would give us confidence that the attacker has been fully evicted and no persistence remains?
✓
Communicate restoration timeline and status to affected internal stakeholders — engineering, product, legal, executive, and customers if required
MED
💬 Discussion Prompt
At what point in this incident did executive leadership and legal become aware? If the answer is "not yet" or "when the deployment freeze was noticed," do we have a pre-defined escalation threshold — e.g., confirmed data exfiltration, production impact, or customer data in scope — that automatically triggers notification up the chain, rather than leaving that call to the IR team?

P-05 POST-INCIDENT & HARDENING IMPROVE ▼
Communications & Compliance
✓
Determine regulatory notification obligations — GDPR 72-hour window, state breach notification laws, SEC cyber disclosure, sector-specific requirements (PCI-DSS, HIPAA)
CRITICAL
💬 Discussion Prompt
We may not yet know for certain whether personal data was exfiltrated — only that the malicious code had access to systems that process it. Under GDPR Article 33, the clock starts when we become "aware" of a breach, not when it's confirmed. At what point in this scenario did we cross that threshold, and is our legal team aligned with that interpretation?
✓
Share IOCs and TTPs with relevant threat intelligence sharing communities (FS-ISAC, H-ISAC, CISA, internal peer network)
MED
💬 Discussion Prompt
Does our organisation have legal clearance and a defined process for sharing incident details externally — including which data can be shared, in what format (STIX/TAXII, TLP markings), and who has authority to approve the disclosure? Have we considered that sharing too early, before legal review, could expose us to liability or breach NDA obligations with third parties?
✓
Complete a formal incident report documenting: timeline, root cause, systems affected, data accessed, actions taken, and business impact
HIGH
💬 Discussion Prompt
Who is the authoritative owner of the incident report, and how do we handle conflicting accounts from different teams about the timeline or decisions made under pressure? Is there a process to reconcile discrepancies before the report is finalised — and is the report stored somewhere an attacker with our access couldn't modify it after the fact?
Process & Control Improvements
✓
Implement or enforce SBOM generation for all builds — make it a mandatory CI artifact alongside build logs
Formats: SPDX, CycloneDX. Integrate with Dependency-Track or similar for continuous monitoring.
HIGH
💬 Discussion Prompt
Generating an SBOM is only half the value — the other half is actively monitoring it. If a new vulnerability is disclosed tomorrow for a package we ingested six months ago, do we have a process that cross-references that CVE against our historical SBOMs and surfaces which currently running services are affected? Or does that analysis only happen manually at incident time?
✓
Enforce package version pinning with integrity verification (hash-pinning / subresource integrity) across all package managers
Require --integrity checks in npm, use hashes in pip requirements.txt, enforce go.sum verification.
HIGH
✓
Stand up or tune a private artifact proxy (Artifactory, Nexus, AWS CodeArtifact) to vet and cache approved package versions — eliminate direct internet pulls in builds
DISCUSSION: Do we have an approved package vetting process? Who owns approvals and how fast can we approve a new package?
HIGH
✓
Integrate SCA tooling into CI pipelines as a blocking gate — fail builds on high/critical vulnerabilities in new or changed dependencies
HIGH
💬 Discussion Prompt
SCA blocking gates create friction that engineering teams are incentivised to work around — suppressing findings, adding exceptions, or bypassing the check to hit a deadline. What governance do we have over exception approvals? Is there a risk of exception fatigue normalising the dismissal of genuine threats?
✓
Apply principle of least privilege to CI/CD pipeline service accounts — build agents should not have deploy or secrets access beyond what is required
HIGH
💬 Discussion Prompt
In this incident, what was the most privileged credential accessible to the compromised build environment? If that credential had been used maliciously, what is the worst thing an attacker could have done with it — and is that worst-case outcome acceptable given the current privilege level? Use this to drive a concrete least-privilege remediation discussion.
✓
Enable runtime behavioural monitoring on build agents and production hosts — alert on unexpected outbound connections or process spawning from package manager paths
MED
💬 Discussion Prompt
Runtime monitoring on build agents generates significant noise — every install, compilation, and test run touches the filesystem and network. How do we baseline "normal" build behaviour so that anomalous activity by a malicious package stands out, rather than being lost in the volume of legitimate build events?
✓
Evaluate namespace reservation for internal package names on public registries to prevent dependency confusion attacks
Reserve your org's internal package names on npm, PyPI, etc. even if you never publish them publicly.
MED
💬 Discussion Prompt
Do we have a current, authoritative list of every internal package name used across all teams? Without that inventory, namespace reservation is incomplete — there may be internal packages we're unaware of that remain vulnerable. Who is responsible for maintaining that list, and how would we know if a new internal package name was introduced without being reserved on public registries?
💬 Discussion Prompt
On a scale of 1–5, how mature is our supply chain security program today? What is the single highest-value control we could implement in the next 30 days? Who is the DRI?
✓
Conduct a blameless post-mortem and document lessons learned — distribute action items with owners and deadlines
HIGH
💬 Discussion Prompt
Looking back at this exercise: which action items from today will actually get completed, and which will stall under normal delivery pressure? What accountability mechanism — executive sponsorship, a security roadmap OKR, or a recurring review — ensures these improvements are tracked to closure rather than documented and forgotten?
